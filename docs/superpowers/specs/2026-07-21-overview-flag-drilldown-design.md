# Overview flag drill-down (server path) — design

Date: 2026-07-21
Status: approved, implementing

## Problem

The server-backed per-submission Overview tab (`views/submission/Overview.tsx`)
renders flags as inert rows: a severity chip, the raw `heuristic_id`, and a
confidence percentage. The `/local` Overview (`views/overview/OverviewView.tsx`)
renders the same flags through `FlagDashboardPanel` → `HeuristicDetailDrawer`,
where each row opens a drawer with the flag's title, description, detail JSON,
and a list of supporting events, each with "jump to raw timeline" and "jump to
replay" buttons.

Reviewers working the real (server-backed) flow therefore cannot get from a flag
to the evidence for it. They have to switch to the Timeline tab and hunt.

Separately, the server path is **wrong** for submissions with more than one
session, in a way that fails silently.

## Root cause of the multi-session breakage

Three facts:

1. `flags.supporting_seqs` stores **globalIdx** ints — session-agnostic,
   unique across the whole submission. They are translated at ingest from the
   analysis-core `${sessionId}:${localSeq}` keys via `index.bySeq`
   (`services/heuristics/run-per-submission.ts`).
2. `flags.session_id` is set to `''` whenever a flag's supporting events span
   more than one session (same file). It is a deep-link convenience, never a
   key.
3. `views/submission/Replay.tsx:toFlag()` reconstructs seqKeys as
   `` `${row.session_id}:${seq}` ``. For a cross-session flag that produces
   `":4880"`, which never resolves in `index.bySeq`, so `JumpControls`'
   "next flag" silently does nothing. For single-session flags it works only
   because the client index built from server rows happens to key `bySeq` as
   `${sessionId}:${globalIdx}`.

**Invariant this design adopts:** on the server path a supporting seq is
resolved by **globalIdx alone**. `session_id` is used for display, never for
resolution.

## Non-goals

- Clickable validation rows on the server path. `validation_results.detail`
  does carry `supportingSeqs`, but those are **session-local** seqs and the
  events API exposes only globalIdx, so resolving them needs an additive
  session-local seq field on `EventRow`. Deferred to its own change.
- Any change to the log format, the hash chain, or the ingest pipeline stage
  ordering.

## Change A — server + contract

### A1. Persist flag prose

`Flag.title` / `Flag.description` are per-instance prose generated by the
heuristics (`"Large paste in hw.py"`), derived from the same data that lands in
`detail`. They are not persisted, so the server path has nothing to show.

Migration `0020_flags_title_description.sql`:

```sql
ALTER TABLE flags ADD COLUMN title       text NOT NULL DEFAULT '';
ALTER TABLE flags ADD COLUMN description text NOT NULL DEFAULT '';
```

Hand-authored SQL, per repo convention (the Drizzle `meta/` snapshots are
intentionally incomplete). Populated by the two places that build flag insert
rows — `services/heuristics/run-per-submission.ts` and
`services/scoring/recompute-submission.ts` — both of which already hold the
in-memory `Flag`.

**Backfill: none.** Pre-migration rows carry `''` and the client falls back to
`heuristic_id`. A tuning recompute repopulates them naturally.

### A2. Expose them

`FlagRowSchema` gains `title: z.string()` and `description: z.string()`.
`services/submissions/flags.ts` selects and returns them.
`openapi/spec/components.ts` is updated in the same diff — the HTTP API shape is
a contract, and both ends move together.

### A3. Validation check labels

`runAndStoreValidation` writes `report.checks` verbatim into
`validation_results.detail`, so each check's human `label` is **already stored**.
`services/submissions/validation.ts` narrows it away in `ValidationCheckRow`.
Widening that type and the schema to carry `label` is a pure passthrough: no
migration, no recompute, correct on every existing row immediately.

The server Overview currently prints raw ids like `monotonic_wall`; it will
print "Monotonic wall clock".

### A4. Session metadata on the summary

`getSubmissionSummary` already calls `loadSubmissionIndex` to derive
`session_ids` and discards the index. Return a richer shape alongside it:

```ts
sessions: { session_id: string; started_at: string; event_count: number }[]
```

`started_at` from `bundle.sessions[i].firstEvent.wall`, `event_count` from
`index.bySessionId`. No extra query and no extra blob parse. `session_ids` stays
for compatibility (`Replay.tsx` reads it).

This is what lets the Overview sessions card be useful without the client paging
the whole event stream.

## Change B — analyzer UI

### B0. Navigation substrate

- `data/useFullEventIndex.ts` gains an `enabled` option so a consumer can defer
  the (expensive, fully-paged) index build until it is actually needed.
- `views/timeline/TimelineInner.tsx` deep-link parser learns a **bare-number**
  `?seq=` form meaning "match `event.seq`". `/local` always emits
  `sessionId:seq`, so there is no ambiguity between the two forms.
- `views/submission/Replay.tsx`:
  - `toFlag()` resolves globalIdx → event → `` `${event.sessionId}:${event.seq}` ``
    so the reconstructed seqKeys actually resolve in `index.bySeq`. Fixes
    "next flag" for cross-session flags.
  - when `?event=` is present and `?session=` is absent or invalid, derive the
    session from the index entry at that globalIdx instead of falling back to
    `sessionIds[0]`.

### B1. One flag panel, two routes

`views/overview/FlagDashboardPanel.tsx` and `HeuristicDetailDrawer.tsx` become
prop-driven and are mounted from both routes — the pattern `TimelineInner`
already establishes. No new directory, no file moves.

New `views/overview/flag-view.ts`:

```ts
export type SupportingRef = {
  id: string;                   // React key + data-testid
  globalIdx: number;            // always known — enough to navigate
  event: IndexedEvent | null;   // null while the index loads / if absent
};

export type FlagView = {
  id, heuristic, title, description, severity, confidence,
  detail?: Record<string, unknown>,
  supporting: SupportingRef[],
};
```

Each route builds `FlagView[]` from its own source:

- `/local` resolves `flag.supportingSeqs` (seqKeys) through `index.bySeq`.
- server resolves `row.supporting_seqs` (globalIdx ints) through a memoized
  `Map<number, IndexedEvent>` keyed by `event.seq`.

This removes the fake-seqKey construction at its root, which is the actual
multi-session bug.

Robustness note: on a server-built index `event.seq` is the server's globalIdx
while `event.globalIdx` is the positional index `i`. They should coincide, but
resolution is by `.seq` and navigation uses the resolved event's own
`.sessionId` / `.globalIdx`, so it stays correct if they ever diverge.

### B2. Lazy index on Overview

`views/submission/Overview.tsx` holds `needsIndex` state, flipped by the panel's
`onDrawerOpen`. Overview costs nothing extra on load — the default tab keeps its
current latency — and the query is cached and shared with the Timeline and
Replay tabs.

### B3. Jumps work before the index lands

Because globalIdx alone is sufficient:

- Timeline: `?tab=timeline&seq=<globalIdx>`
- Replay: `?tab=replay&event=<globalIdx>` (session derived on arrival)

So the drawer's jump buttons are live immediately, even while supporting-event
metadata is still loading.

### B4. The drawer, multi-session

Supporting events group under session headers (`Session 2 · <start time>`), with
a `spans N sessions` marker when the flag crosses sessions. Unresolved rows
render as `event #4880` with both jump buttons still enabled.

### B5. Sessions card

New `views/submission/SessionsCard.tsx`, driven entirely by `summary.sessions`
from A4. Rendered only when `sessions.length > 1`, mirroring
`SummaryStatsPanel`'s rule on the `/local` side. Row click →
`?tab=replay&session=<id>`.

## Commit split

Change B touches 9 files, over the ~200-line / ~5-file guideline, so the work
lands as three commits on one branch:

1. **A** — server + contract (migration, ingest/recompute population, flags and
   validation services, shared schemas, OpenAPI).
2. **B0** — navigation substrate. Independently testable; fixes the
   multi-session jump bug on its own.
3. **B1** — the panel, both wirings, sessions card, validation labels.

## Testing

- `run-per-submission` / `recompute-submission`: assert `title` / `description`
  are persisted from the in-memory flag.
- `services/submissions/flags`: assert the new columns round-trip.
- `services/submissions/validation`: assert `label` passes through.
- `services/submissions/summary`: assert `sessions[]` shape, ordering, and
  counts on a two-session bundle.
- `flag-view.ts`: unit tests for both resolvers, including a cross-session flag
  and an unresolvable globalIdx.
- `TimelineInner`: bare-number deep link selects the right event; the
  `sessionId:seq` form still works.
- `Replay`: `toFlag` produces resolvable seqKeys for a cross-session flag;
  `?event=` without `?session=` lands in the correct session.
- `Overview` (server): drawer opens, does not fetch the index until it does,
  jump buttons emit the right search params.
