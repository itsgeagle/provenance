# Provenance Heuristics

The Provenance Analyzer's heuristics suite reads a sealed `.slog` bundle and produces a list of `Flag`s shown in the Submission Overview and Replay views. Every flag is a sort-priority signal for staff triage — never a verdict. The replay UI is the verification path; clicking any flag jumps to the supporting event in context.

For the overall product, see the [repo root README](../README.md). For the full spec, see [`prd.md`](./prd.md). For the analyzer build plan, see [`analyzer-implementation-plan.md`](./analyzer-implementation-plan.md).

## How heuristics are run

A single entry point — [`runHeuristics`](../packages/analyzer/src/heuristics/run-heuristics.ts) — iterates a fixed registry, collects all flags, appends [integrity flags](../packages/analyzer/src/heuristics/integrity-flags.ts) derived from the bundle validation report (PRD §5.4), and sorts:

1. severity descending (`high` → `medium` → `low` → `info`)
2. confidence descending
3. supporting seq key ascending (stable tie-break — deterministic across runs and snapshot-test-friendly)

Cross-submission heuristics run separately via [`runCrossHeuristics`](../packages/analyzer/src/heuristics/cross/run-cross-heuristics.ts), only when more than one bundle is loaded into the `/compare` view.

Configuration defaults live in [`config.ts`](../packages/analyzer/src/heuristics/config.ts). Course staff can override per-heuristic via the `HeuristicConfig` shape passed to `runHeuristics`.

Severity / confidence semantics:

- **Severity** is the heuristic's claim about how serious the signal is in isolation: `high`, `medium`, `low`, `info`. `info` is informational context (e.g. "an AI extension was active"), not an integrity claim by itself.
- **Confidence** is how reliable the signal is on this particular bundle (0.0–1.0). A high-severity flag with 0.6 confidence is "loud signal, less sure"; a medium with 0.95 confidence is "subtle but reliable". Both are surfaced; staff decide weighting.

**Shared iterator.** The three paste-content heuristics (`large_paste`, `paste_is_solution`, `paste_matches_known_source`) iterate paste-shaped events through one helper, [`candidate-pastes.ts`](../packages/analyzer/src/heuristics/candidate-pastes.ts). The helper unifies native `paste` events and recorder-v1.2 `doc.change` events with `source: "paste_likely" | "paste_confirmed"`, yielding one `CandidatePaste` per inserted blob. When a new paste source is introduced (drag-and-drop file insertion, snippet expansion, etc.) only this file needs updating.

The validation report is produced by [`run-validation.ts`](../packages/analyzer/src/validation/run-validation.ts); the adapter that lifts failed checks into the flag list lives in [`integrity-flags.ts`](../packages/analyzer/src/heuristics/integrity-flags.ts).

| Heuristic                   | Severity | Detects                                                                                        | Source check                                                                                 |

| `monotonic_t_regression`    | medium   | Session-relative `t` field went backwards.                                                     | [`verify-monotonic-t.ts`](../packages/analyzer/src/validation/verify-monotonic-t.ts)         |

## Things to keep in mind for staff communication

- **None of these are verdicts.** Per PRD §7.4: "the score is never the verdict — it's a sort order for staff triage." Every escalation goes to a human reviewer who verifies via the replay UI.
- **Process evidence over content classification.** The system intentionally does not run any "is this code AI-generated?" classifier. All signals are about _how_ the code came into existence — pastes, external edits, timing — not about the code itself. This is defensible in an academic integrity hearing in a way that statistical AI-detection scores are not (PRD §1, §2 NG5).
- **Known-good corpus is empty by default.** `paste_matches_known_source` will not fire until course staff populate the corpus with leaked solutions, common Stack Overflow answers, or whatever else they want to match against. Mechanism is in place; content is staff's call (PRD §10 Q4).
- **Cross-submission heuristics require a batch.** They don't run on single-submission review. Staff load multiple bundles into `/compare` to surface `paste_shared_across_students` and `editing_pattern_clone`.
- **Every flag has a "Jump to replay" link.** No claim is shown in the analyzer without a way to verify it against the actual event stream in one click.
- **Configuration is in code.** Default thresholds are in [`config.ts`](../packages/analyzer/src/heuristics/config.ts). Tuning is staff's call once a labeled sample exists; PRD §10 Q5 flagged this as an open question.
